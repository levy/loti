# Implementation

> **Note:** this describes the pre-refactor monolithic simulation; the protocol logic now lives
> in [`src/core/`](../src/core) behind ports, hosted by [`src/app/sim/`](../src/app/sim)
> (simulation) and [`src/app/lotid`](../src/app/lotid) (the production node) — see
> [architecture.md](architecture.md). The file links below point to the code that replaced each
> original module.

This document describes **how the current LOTI code works**. It is an OMNeT++ 5.4 / INET 4.0
simulation model, not a production node. For the theory it realizes see
[theory.md](theory.md); for the example network and results see
[simulation.md](simulation.md); for the exact list of what is and is not implemented see
[paper-vs-implementation.md](paper-vs-implementation.md).

All source lives in [`src/`](../src); the runnable example lives in [`sim/`](../sim).

## Module map

Each simulated host (`inet.node.inet.StandardHost`) runs three LOTI application modules on
top of a normal INET UDP/IPv4/Ethernet stack. A single network-wide helper module builds
the peer-to-peer overlay.

| Module | Kind | Role |
| --- | --- | --- |
| [`Daemon`](../src/app/sim/Daemon.cpp) | per-host app (`app[0]`) | The protocol engine: grows the local clock-event DAG, talks to neighbors over UDP, runs the three discoveries. |
| [`Publisher`](../src/app/sim/Publisher.cpp) | per-host app (`app[1]`) | Periodically creates events with random content via the local daemon. |
| [`Browser`](../src/app/sim/Browser.cpp) | per-host app (`app[2]`) | Periodically picks random event(s) and asks the local daemon to run a discovery. |
| [`NetworkConfigurator`](../src/app/sim/NetworkConfigurator.cpp) | one per network | Builds the overlay neighbor table and next-hop routing table from the physical topology. |
| `ResultFilter` (removed; see [`src/adapters/sim/telemetry.hpp`](../src/adapters/sim/telemetry.hpp)) | registered filters | Custom `cResultFilter`s that turn emitted discovery objects into scalar statistics. |

The three apps all declare `like IApp` and connect to the host's UDP layer through
`socketIn`/`socketOut` gates, but only the `Daemon` actually opens a socket. `Publisher` and
`Browser` reach the daemon by a direct C++ method call (`getModuleFromPar<Daemon>`), not over
the network — they are co-located control clients of the daemon.

## Data model

The wire/stored data structures are generated by the OMNeT++ message compiler from
[`Data.msg`](../src/core/domain/types.hpp), with primitive typedefs in [`Type.msg`](../src/core/domain/types.hpp):

```
NodeId    = uint64_t          // a node's identity (here: the daemon module's getId())
Salt      = uint64_t          // random per-event salt
EventHash = vector<uint8_t>   // a 32-byte SHA-256 digest
```

| Type | Fields | Notes |
| --- | --- | --- |
| `EventReference` | `creator: NodeId`, `hash: EventHash` | A typed pointer to another event/clock event. |
| `Event` | `creator`, `hash`, `data: ByteVector`, `salt`, `referencedEvents: [EventReference]` | A published event. `data` is the content. |
| `ClockEvent` | `creator`, `hash`, `timestamp: simtime_t`, `salt`, `referencedEvents` | A timestamped event. |
| `LocalClockEvent` : `ClockEvent` | + `referencingEvents: [EventReference]` | A clock event **created by this node**; also tracks who references it (reverse edges). |
| `EventChain` | `event: Event`, `lowerBound: deque<ClockEvent>`, `upperBound: deque<ClockEvent>` | The proof object: a hash chain enclosing `event`. |
| `Neighbor` | `nodeId`, `address: Ipv4Address`, `lastClockEventHash` | An overlay neighbor and the newest clock event of theirs we know. |
| `EventChainDiscovery` | `startTime`, `endTime`, `originator`, `event`, `chain`, `state` | Originator-side record of a chain discovery. |
| `EventBoundsDiscovery` | `startTime`, `endTime`, `event`, `lowerBound: simtime_t`, `upperBound: simtime_t`, `state` | Result of a bounds discovery. |
| `EventOrderDiscovery` | `startTime`, `endTime`, `event1`, `event2`, `order: int`, `state` | Result of an order discovery. |

`DiscoveryState` is one of `DS_DISCOVERY_INPROGRESS`, `DS_DISCOVERY_COMPLETED`,
`DS_DISCOVERY_ABORTED`. `order` is `-1` (event1 before event2), `+1` (after), or `0`
(undetermined). `EventChainDiscoveryParticipation` is declared but unused.

### The clock-event DAG, concretely

An `Event` embeds the hash of the last local clock event (a backward edge into the clock
chain). A `LocalClockEvent` embeds three kinds of backward edges in its `referencedEvents`:

1. the **previous local clock event** (extends this node's own chain),
2. the **latest known clock event of each neighbor** (cross-links into neighbor DAGs),
3. **all local events created since the last clock event** (pins those events in time).

`referencingEvents` on a `LocalClockEvent` holds the **reverse** cross-links: which neighbor
clock events point back at this one. These reverse edges are learned lazily from neighbor
notifications (see below) and are what makes the *upper* bound walkable.

## Hashing

Hashing uses the bundled header-only [`picosha2.hpp`](../src/core/hash/picosha2.hpp) (SHA-256). Digests are
`picosha2::k_digest_size` = 32 bytes. Serialization for hashing is explicit and canonical
(`Daemon::calculateEventHash` / `calculateClockEventHash`):

- **Event**: `data` bytes, then `salt` (big-endian u64), then for each referenced event its
  `creator` (u64) and `hash` bytes.
- **Clock event**: `timestamp.raw()` (u64), then `salt`, then each referenced event's
  `creator` and `hash`.

The salt is 64 random bits (`generateSalt` composes two `intuniform` draws). Because the
hash covers the referenced hashes, every reference is a cryptographic commitment to the
referenced object's entire history — this is the property the security argument in
[theory.md](theory.md) relies on.

## Daemon: local state

`Daemon` holds (see [`Daemon.h`](../src/app/sim/Daemon.h)):

- `nodeId` — this daemon's identity (`getId()`).
- `neighbors: map<NodeId, Neighbor>` — the overlay neighbor set.
- `allEvents: vector<Event>` — every event this node published.
- `unreferencedEvents: vector<Event>` — events not yet pinned by a clock event.
- `allClockEvents: vector<LocalClockEvent>` — this node's clock-event chain, in creation
  order (so index order == time order).
- `destinationToNextHop: map<NodeId, NodeId>` — overlay routing table.
- `eventHashToEventIndex`, `eventHashToClockEventIndex` — hash → index lookups.
- `eventHashToReferencingEventIndex: multimap<EventHash, unsigned>` — for an event/clock-event
  hash, the index(es) of the **local clock event(s) that reference it** (forward lookup used
  to grow the upper bound).
- `eventChainDiscoveries`, `eventBoundsDiscoveries`, `eventOrderDiscoveries` — in-flight and
  finished discoveries, each paired with the list of callbacks waiting on it. Chain
  discoveries are keyed by event hash; order discoveries by the `(hash1, hash2)` pair.
- a `UdpSocket` bound to **port 666**, plus two self-message timers.

Initialization uses INET init stages: local state at `INITSTAGE_LOCAL`, the socket at
`INITSTAGE_APPLICATION_LAYER`. The overlay is filled in by `NetworkConfigurator` at
`INITSTAGE_NETWORK_LAYER_3` (it is a `friend` of `Daemon` and writes `neighbors` /
`destinationToNextHop` directly).

## Growing the local DAG

### Publishing an event

`Publisher::processCreateEventTimer` fires every `createEventInterval` and calls
`Daemon::publishEvent(data)` with `contentLength` random bytes. `insertEvent`:

1. references the last local clock event (if any),
2. sets `creator = nodeId`, a random salt, computes the hash,
3. appends to `allEvents`, indexes it, and pushes it onto `unreferencedEvents`,
4. emits `eventCreated`.

### Creating a clock event

`Daemon::processCreateClockEventTimer` fires every `createClockEventInterval`. `insertClockEvent`
builds `referencedEvents` = {last local clock event} ∪ {each neighbor's `lastClockEventHash`}
∪ {all `unreferencedEvents`}, stamps `timestamp = simTime()`, salts, hashes, and appends to
`allClockEvents`. Then the timer handler:

1. indexes the new clock event by hash,
2. for each referenced event, records the reverse edge in `eventHashToReferencingEventIndex`,
3. **clears `unreferencedEvents`** (they are now pinned),
4. emits `clockEventCreated`,
5. sends a **clock event notification** to every neighbor.

### Clock event notifications

`sendClockEventNotification(neighbor, clockEvent)` sends a UDP packet carrying
`(lastClockEventHash = my new clock event, neighborLastClockEventHash = the newest clock
event of *that* neighbor I currently know)`.

`processClockEventNotification` on the receiver:

1. updates `neighbor.lastClockEventHash` to the sender's new clock event,
2. if `neighborLastClockEventHash` names one of **my** clock events, appends a reverse edge
   to that local clock event's `referencingEvents`: `(creator = sender, hash = sender's new
   clock event)`.

Only the **hash** of a clock event is ever broadcast — never its contents, and never events.
This is the "clock event hashes are distributed to neighbors; events are not distributed"
property from the paper.

## The overlay: NetworkConfigurator

INET's `Ipv4NetworkConfigurator` assigns addresses and installs shortest-path IPv4 routes.
LOTI's own `NetworkConfigurator` then builds the **application-level overlay** at
`INITSTAGE_NETWORK_LAYER_3`:

- It extracts the topology of `@networkNode` modules, and for every destination computes
  unweighted single-shortest-paths.
- For each `(source → destination)` pair it records `source.destinationToNextHop[dest] =
  firstHop`, and registers the first hop as a `Neighbor` of the source (with the hop's IPv4
  address), and vice versa.

In the bundled example the overlay neighbor graph therefore coincides with the physical
topology (neighbors are directly connected). The paper's intent is that this second layer
could describe a different, non-physical overlay; the code supports that shape but the
example does not exercise it.

## Protocol packets

Three UDP message types (see [`Packet.msg`](../src/core/wire/packets.hpp)), each prefixed by a
`LotiHeader { type, neighbor }` where `neighbor` is the **sender's** `NodeId` (used by the
receiver to look the sender up in its `neighbors` map):

| `LotiType` | Payload | Direction |
| --- | --- | --- |
| `LT_CLOCK_EVENT_NOTIFICATION` | `ClockEventNotification { lastClockEventHash, neighborLastClockEventHash }` | broadcast to each neighbor on every clock event |
| `LT_EVENT_CHAIN_DISCOVERY_REQUEST` | `EventChainDiscoveryRequest { originator, event: EventReference }` | routed toward the event's creator |
| `LT_EVENT_CHAIN_DISCOVERY_RESPONSE` | `EventChainDiscoveryResponse { originator, chain: EventChain }` | routed back toward the originator |

`socketDataArrived` pops the header, verifies the sender is a known neighbor, then dispatches
by type. Packet chunk lengths are computed by the `calculate*Size` helpers in
[`Data.cc`](../src/core/hash/hashing.cpp) so that byte-accounting statistics are meaningful (event `data`
content is deliberately **excluded** from the accounted size).

## Event chain discovery — the core algorithm

An event chain discovery reconstructs an `EventChain` whose concatenation
`lowerBound · event · upperBound` is a single hash chain that **begins and ends at the
originator's own clock events** and passes through the requested `event`. This is the proof
object described in [theory.md](theory.md).

> The algorithm below is the **width-1 special case**: one deterministic walk along the routing
> shortest path. How it generalizes to the paper's probabilistic beam search — and how that
> copes with a churning, lazily-linked network — is covered in
> [dynamic-discovery.md](dynamic-discovery.md).

### Starting (originator)

`Daemon::discoverEventChain(event, callback)`:

- If a discovery for `event.hash` already exists, the call **coalesces**: attach the callback
  (in progress), or immediately replay the aborted/completed result.
- Otherwise create an `EventChainDiscovery` (state `INPROGRESS`) and:
  - **If this node is the creator**, build the chain locally — `addLocalLowerBound` +
    `addLocalUpperBound` — and complete immediately (no packets).
  - **Otherwise** look up `findNextHopNeighbor(event.creator)` and send an
    `EventChainDiscoveryRequest` with `originator = this node`.

Only the **originator** keeps an `EventChainDiscovery` record. Intermediate and creator nodes
handle request/response packets statelessly, building an `EventChain` on the stack and
forwarding it.

### Routing the request

`processEventChainDiscoveryRequest`:

- **Not the creator** → forward the request to `findNextHopNeighbor(event.creator)`.
- **Is the creator** → seed the chain: `setEvent`, then in order
  `addLocalLowerBound`, `addLocalUpperBound`, `extendLowerBoundForNeighbor(sender)`,
  `extendUpperBoundForNeighbor(sender)`; if all succeed, send an
  `EventChainDiscoveryResponse` back to the neighbor the request came from. Any failing step
  logs a warning and drops the discovery (which will later expire at the originator).

### The four chain-building primitives

These are the heart of the algorithm (all in [`Daemon.cc`](../src/core/node.cpp)):

- **`addLocalLowerBound(chain)`** — look at the `referencedEvents` of the current front of
  `lowerBound` (or of `event` if `lowerBound` is empty); find one created by *this* node,
  resolve it to a local clock event, and `push_front` it. This prepends the local clock event
  that the current front *references* (a step backward/older).

- **`addLocalUpperBound(chain)`** — take the hash of the current back of `upperBound` (or of
  `event` if empty), look it up in `eventHashToReferencingEventIndex`, and `push_back` the
  first local clock event that *references* it (a step forward/newer).

- **`extendLowerBoundForNeighbor(neighbor, chain)`** — starting from the front local clock
  event, walk **backward** through `allClockEvents` (`index--`, `push_front` each), stopping
  when a clock event whose `referencedEvents` include one created by `neighbor` reaches the
  front. That front now cross-links into the neighbor's DAG, so the neighbor can continue the
  chain. Returns `false` if the start of the chain is reached first.

- **`extendUpperBoundForNeighbor(neighbor, chain)`** — the mirror image: walk **forward**
  (`index++`, `push_back`) until a local clock event whose `referencingEvents` include one
  created by `neighbor` reaches the back. Returns `false` if the end of the chain is reached
  first.

The two `extend*` primitives are why a node "takes care of providing enough local clock
events for the next neighbor" (README): it splices its own chain far enough that both
endpoints touch the next node's clock events.

### Routing the response back

`processEventChainDiscoveryResponse`:

- **Not the originator** (intermediate node) → `addLocalLowerBound`, `addLocalUpperBound`
  (splice this node's clock events onto the ends the previous node left touching it), then
  `extendLowerBoundForNeighbor(nextHop)`, `extendUpperBoundForNeighbor(nextHop)` to prepare
  for the next hop, then forward toward `findNextHopNeighbor(originator)`.
- **Is the originator** → `addLocalLowerBound`, `addLocalUpperBound` to attach the
  originator's own clock events at both ends, then `completeEventChainDiscovery`. Any failing
  step aborts the discovery.

The net effect: the request walks hop-by-hop to the creator; the response walks hop-by-hop
back; at each node the chain accretes that node's local clock events; and it arrives at the
originator with the originator's own clock events at both ends.

### Completing and validating

`completeEventChainDiscovery` sets state `COMPLETED`, then **always** calls
`validateEventChainDiscoveryResult` before invoking callbacks:

- `validateEventChainDiscoveryResult` requires `lowerBound.front().creator == nodeId` and
  `upperBound.back().creator == nodeId` — the endpoints must be the originator's clock events
  (so the bounds are in the originator's local time) — then calls `validateEventChain`.
- `validateEventChain` verifies, over `lowerBound · event · upperBound`:
  - every clock event's stored hash equals a freshly recomputed hash,
  - every element after the first embeds a reference to its **immediate predecessor**
    (the tamper-evident hash-chain property),
  - the event references the last lower-bound clock event (unless the lower bound is empty).

Per-step consistency is additionally asserted during construction via
`ASSERT_VALID_EVENT_CHAIN`, but that macro is compiled out in release builds (`NDEBUG`); the
final `validateEventChainDiscoveryResult` always runs.

## Event bounds discovery

`discoverEventBounds(event, callback)` coalesces like chain discovery, then simply starts an
event chain discovery with the **daemon itself** as the chain callback
(`discoverEventChain(event, *this)`). When that chain completes,
`Daemon::processEventChainDiscoveryCompleted` fills the matching `EventBoundsDiscovery` with
`lowerBound = chain.lowerBound.front().timestamp` and
`upperBound = chain.upperBound.back().timestamp` and completes it. The bounds are thus the
timestamps of the originator's two enclosing clock events. (The paper notes this could be
made more robust by running several chain discoveries; the code runs exactly one.)

## Event order discovery

`discoverEventOrder(event1, event2, callback)` coalesces on the pair, then starts **two**
independent chain discoveries, one per event. `processEventChainDiscoveryCompleted` notices
when both underlying chains are complete and calls `compareEventChains`:

```
if  chain1.upperBound.back().timestamp < chain2.lowerBound.front().timestamp  ->  -1
elif chain2.upperBound.back().timestamp < chain1.lowerBound.front().timestamp  ->  +1
else                                                                            ->   0
```

Because both chains are anchored in the **same originator's** clock events, the comparison is
a comparison of that node's local timestamps: if event1's whole interval lies strictly before
event2's (or vice versa) the order is definite; if the intervals overlap the order is
undetermined (`0`). A single completed chain discovery can satisfy a bounds discovery and an
order discovery at once, since all three share the `eventChainDiscoveries` map.

## Discovery lifecycle and expiry

Each discovery type has `insert*` / `abort*` / `complete*` helpers that flip `state`, stamp
`endTime`, notify every registered callback, and emit a `*Started` / `*Aborted` /
`*Completed` signal.

`purgeDiscoveriesTimer` fires every `discoveryExpiryTime / 100`. Any discovery whose
`startTime` is older than `discoveryExpiryTime` is removed; if it was still `INPROGRESS` it is
first aborted (which fires the callbacks and the `*Aborted` signal). Aborting an event chain
discovery cascades: `processEventChainDiscoveryAborted` also aborts the dependent bounds and
order discoveries. Expiry is the main reason discoveries fail; the other is a chain that
cannot be closed because the necessary reverse edges (`referencingEvents`) have not yet been
learned from neighbor notifications — which is why the first and last events created near the
start of a run often cannot be bounded (see [simulation.md](simulation.md)).

## Applications

- **`Publisher`** — one timer (`createEventInterval`); each firing publishes an event of
  `contentLength` random bytes through the local daemon.
- **`Browser`** — three independent timers (`discoverEventChainInterval`,
  `discoverEventBoundsInterval`, `discoverEventOrderInterval`); an interval of `0s` disables
  that timer. Each firing picks a random event — via the `NetworkConfigurator`, it selects a
  random network node and a random event from **that node's** daemon (an out-of-band "oracle"
  standing in for the web servers of a real deployment) — and requests the corresponding
  discovery. Results are logged (`EV_INFO`); the browser implements all three discovery
  callback interfaces.

## Statistics

Signals are declared on the `Daemon` in [`Daemon.ned`](../src/app/sim/Daemon.ned) and aggregated at
the network level in [`sim/Network.ned`](../sim/Network.ned):

- **Counts** — clock/event creation counts and started/aborted/completed counts for each of
  the three discovery types.
- **Vectors** — `clockEventsFileLength` / `eventsFileLength` accumulate `clockEventSize` /
  `eventSize` over time (per-node storage growth).
- **Vectors + histograms** — chain discovery time / length / interval, bounds discovery time
  / interval, order discovery time / order.

The derived quantities are produced by the custom `cResultFilter`s in
`ResultFilter.cc` (removed; see [`src/adapters/sim/telemetry.hpp`](../src/adapters/sim/telemetry.hpp)), each registered by name (e.g.
`eventChainDiscoveryTime` = `endTime − startTime`, `eventChainDiscoveryLength` =
`|lowerBound| + |upperBound| + 1`, `eventBoundsDiscoveryInterval` = `upperBound − lowerBound`,
`clockEventSize` / `eventSize` from the byte-accounting helpers). INET additionally records
its own UDP/packet statistics.
